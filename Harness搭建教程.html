Hermes Agent 实战搭建教程

这不是概念文章，而是一份能照着做的搭建手册。目标是把一个刚装好的 Hermes Agent，变成有规则、有记忆、有交接、有备份、有报告输出的可用工作系统。小白不需要先懂 Harness，按八步走就能跑通第一版。

摘要

Harness 可以简单理解为“给 AI Agent 装一套工作安全带和操作台”。它不只是聊天，而是让 Agent 每次都知道：先读什么、怎么做、错了记哪里、完成后怎么交接。

8 步从空文件夹到可用体系
30 分钟能跑通第一版
0 后端全部用文件驱动

最终目录长这样

D:\Codex\workspace\
├── AGENTS.md                  ← Agent 启动就认这个
├── 规则层\
│   ├── 执行规则.md             ← 所有任务的总规则
│   └── 模型规则\
│       └── 阿珂执行脑协议.md    ← 执行脑约束
├── 记忆层\
│   ├── USER.md                ← 你是谁
│   ├── MEMORY.md              ← 环境/工具/事实
│   └── SOUL.md                ← Agent 人格
├── 执行流转层\
│   └── 当前状态.md             ← 双 Agent 交接
├── 报告\
│   └── *.html                 ← 生成的报告
└── 备份脚本                   ← 可选：自动备份

0第〇步：你需要的准备

做什么

确认你已经有基础工具，不然教程后面的文件、GitHub Pages 和自动备份跑不起来。

为什么

Agent 本身只是大脑。workspace、GitHub、备份脚本这些东西，才是让它长期稳定工作的“身体”。

怎么做

# 检查 Git
 git --version

# 检查 Python
 python --version

# 创建主目录
 mkdir D:\Codex
✅ 验证：PowerShell 能输出 Git 和 Python 版本，且 `D:\Codex` 文件夹存在。

1第一步：建 workspace

做什么

创建一个专门给 Agent 读写的工作区，并放入最小版 `AGENTS.md`。

为什么

`AGENTS.md` 是 Agent 的项目级说明书。新对话、换模型、换工具时，它都能先读这个文件，知道这里的规矩。

怎么做

mkdir D:\Codex\workspace
notepad D:\Codex\workspace\AGENTS.md

AGENTS.md 最小模板

# AGENTS.md

- 默认中文回答。
- 非聊天任务先读 D:\Codex\workspace\规则层\执行规则.md。
- 默认只读，不要擅自删除、覆盖、提交或推送。
- 项目文件统一放 D:\Codex\代码存放处\。
- 完成任务后更新 D:\Codex\workspace\执行流转层\当前状态.md。
✅ 验证：重新打开 `AGENTS.md`，能看到这 5 条规则。

2第二步：搭规则层

做什么

创建 `规则层\执行规则.md`，告诉 Agent 接到任务后先做什么、再做什么、什么时候停。

为什么

没有规则层，Agent 容易凭感觉开干。规则层的作用就是把“先读需求、再执行、最后验证”固定下来。

怎么做

mkdir D:\Codex\workspace\规则层
notepad D:\Codex\workspace\规则层\执行规则.md

执行规则最小模板

# 执行规则

## 三条铁律
1. 非聊天任务先读本文件，再动手。
2. 做项目先明确需求；不清楚就问。
3. 改完必须验证，失败最多重试 3 次。

## Skill 扫描
有相关 skill 就先读 skill，再写代码。

## 收尾
完成后更新 当前状态.md，记录做了什么。
✅ 验证：`D:\Codex\workspace\规则层\执行规则.md` 存在，并能读到“三条铁律”。

5第五步：加阿珂执行脑协议

做什么

创建一份专门约束执行风格的协议，让 Agent 不要闷头乱干。

为什么

`AGENTS.md` 管总规则，执行脑协议管具体做事风格：先审后干、敢砍简化、写完必验、成本意识。

怎么做

mkdir D:\Codex\workspace\规则层\模型规则
notepad D:\Codex\workspace\规则层\模型规则\阿珂执行脑协议.md

协议最小模板

# 阿珂执行脑协议

1. 先审后干：读完方案先判断是否可行。
2. 敢砍简化：过度设计直接压成 MVP。
3. 写完必验：代码、配置、文档都要检查。
4. 成本意识：能本地做不用 API。
5. 诚实原则：卡了就说卡在哪里。
6. 状态同步：项目结束更新 当前状态.md。

挂到 AGENTS.md

6第六步：自动备份

做什么

把 workspace 定时备份到 GitHub 私有仓库，防止规则、记忆、状态文件丢失。

为什么

规则层和记忆层一旦丢了，Agent 就会“失忆”。备份是这套系统的保险。

怎么做

# 1. 在 GitHub 创建私有仓库，例如 workspace-backup
# 2. 本地初始化
cd D:\Codex\workspace
git init
git remote add origin https://github.com/你的用户名/workspace-backup.git

# 3. 第一次提交
git add .
git commit -m "Initial workspace backup"
git push -u origin main

任务计划程序

# 每天 00:00 运行备份脚本，示例：
schtasks /create /tn WorkspaceBackup /tr "D:\Codex\备份\backup_workspace.bat" /sc daily /st 00:00
注意：备份前必须排除 `.env`、Token、API Key、聊天记录、数据库和个人隐私文件。
✅ 验证：GitHub 私有仓库能看到最新提交，本地 `git status` 是干净的。

7第七步：HTML 报告模板

做什么

准备一个固定风格的 HTML 报告模板，用来输出项目复盘、教程、分析和交付总结。

为什么

每次报告都重新设计会浪费时间。固定模板能让 Agent 专注内容，也方便 GitHub Pages 发布。

怎么做

mkdir D:\Codex\workspace\报告
notepad D:\Codex\workspace\规则层\报告规则.md
notepad D:\Codex\workspace\规则层\报告模板.html

报告规则最小模板

# 报告规则

- 报告用湖泊色 #2E8B8B。
- 单文件 HTML，CSS/JS 内嵌。
- 左侧导航，正文卡片。
- 完成后用浏览器打开验证。
- 不写入密钥、Token、隐私聊天记录。

部署到 GitHub Pages

# 创建公开仓库 reports
# 把 HTML 文件复制进去后：
git add .
git commit -m "Add report"
git push

# 打开仓库 Settings → Pages → Deploy from branch → main
✅ 验证：浏览器能打开 `https://你的用户名.github.io/reports/xxx.html`。

8第八步：验证整套系统

做什么

不要只看文件有没有创建，要验证 Agent 是否真的会按这套系统工作。

怎么做

# 让 Agent 做一个小任务，例如：
请按 D:\Codex\workspace\规则层\执行规则.md，生成一份今天的任务总结报告。

检查清单

启动能读规则

Agent 开始任务前会提到执行规则或当前状态。

过程能交接

`当前状态.md` 能看出做到哪一步。

错误能沉淀

踩坑会写进记忆或 BugTracker，而不是口头说说。

结果能发布

HTML 报告能本地打开，也能推到 GitHub Pages。
