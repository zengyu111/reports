Hermes Agent 实战搭建教程

这是一份给新手的搭建手册。你不需要先懂 Harness，按八步走：建 workspace、加规则、加记忆、做交接、加协议、做备份、配报告、最后验证。

摘要

Harness 可以简单理解为“给 AI Agent 装一套工作安全带和操作台”。它不只是聊天，而是让 Agent 每次都知道：先读什么、怎么做、错了记哪里、完成后怎么交接。

8 步从空文件夹到可用体系
30 分钟能跑通第一版
0 后端全部用文件驱动

最终目录长这样

D:\workspace\
├── AGENTS.md                  ← Agent 启动就认这个
├── 规则层\
│   ├── 执行规则.md             ← 所有任务的总规则
│   └── 模型规则\
│       └── 执行脑协议.md        ← 执行风格约束
├── 记忆层\
│   ├── USER.md                ← 你是谁
│   ├── MEMORY.md              ← 环境/工具/事实
│   └── SOUL.md                ← Agent 人格
├── 执行流转层\
│   └── 当前状态.md             ← 双 Agent 交接
├── 报告\
│   └── *.html                 ← 生成的报告
└── 备份脚本                   ← 可选：自动备份

Hermes 原生能力 vs 自定义 Harness

Hermes 原生能力本教程自定义 Harness
AGENTS.md 上下文加载Workspace 是唯一真相源
Hermes 自身配置共享 SOUL.md / USER.md / MEMORY.md
Hermes 自身 Memory / Profile / Backup 等机制当前状态.md、双 Agent 文件交接、执行规则与验证规则

本教程不是把所有能力都绑定在 Hermes 内部，而是以 Workspace 作为唯一真相源，让小墨、阿珂以及未来其他 Agent 都能读取同一套共享文件。

注意：SOUL.md、USER.md、MEMORY.md 放在 workspace 的记忆层里是有意设计，不是路径错误；不要移动到 ~/.hermes/。

0第〇步：你需要的准备

做什么

准备 Hermes Agent、一个 AI 编程助手、一个 GitHub 账号，以及一个你准备用来长期放 workspace 的磁盘。

为什么

Agent 是大脑，workspace 是长期记忆和工作台。没有 workspace，换对话后很容易失去上下文。

怎么做

# 检查 Git
 git --version

# 检查 Python
 python --version

# 创建主目录
 mkdir D:
✅ 验证：PowerShell 能输出 Git 和 Python 版本，且 `D:` 文件夹存在。

1第一步：建 workspace

做什么

创建一个专门给 Agent 读写的工作区，并放入最小版 `AGENTS.md`。

为什么

`AGENTS.md` 是 Agent 的项目级说明书。新对话、换模型、换工具时，它都能先读这个文件，知道这里的规矩。

怎么做

mkdir D:\workspace
notepad D:\workspace\AGENTS.md

AGENTS.md 最小模板

# AGENTS.md

- 默认中文回答。
- 非聊天任务先读 D:\workspace\规则层\执行规则.md。
- 默认只读，不要擅自删除、覆盖、提交或推送。
- 项目文件统一放 D:\代码存放处\。
- 完成任务后更新 D:\workspace\执行流转层\当前状态.md。
✅ 自检：先进入 `D:\workspace`，再启动 Agent；确认 `AGENTS.md` 位于这个 workspace 根目录，且文件名不是 `AGENTS.md.txt`。
出错怎么办：这是最高优先级排错项。即使 AGENTS.md 内容完全正确，如果 Agent 不是从正确的 Workspace 目录进入，也可能读不到项目规则。排错顺序：1. 检查当前工作目录 / 启动目录；2. 检查 AGENTS.md 是否位于正确 Workspace；3. 检查文件名和大小写；4. 检查文件内容和编码；5. 检查 AGENTS.md 是否正确指向其他规则文件。
错误：
在其他目录启动 Agent，
再期待它自动理解 D:\workspace 的规则。
正确：
先进入 D:\workspace，
再启动或让 Agent 读取该 Workspace。

2第二步：搭规则层

做什么

创建 `规则层\执行规则.md`，告诉 Agent 接到任务后先做什么、再做什么、什么时候停。

为什么

没有规则层，Agent 容易凭感觉开干。规则层的作用就是把“先读需求、再执行、最后验证”固定下来。

怎么做

mkdir D:\workspace\规则层
notepad D:\workspace\规则层\执行规则.md

执行规则最小模板

# 执行规则

## 三条铁律
1. 非聊天任务先读本文件，再动手。
2. 做项目先明确需求；不清楚就问。
3. 改完必须验证，失败最多重试 3 次。

## Skill 扫描
有相关 skill 就先读 skill，再写代码。

## 收尾
完成后更新 当前状态.md，记录做了什么。
✅ 自检：`规则层\执行规则.md` 存在，并能读到“三条铁律”。
出错怎么办：Agent 跳过规则时，把“非聊天任务先读 规则层\执行规则.md”放到 AGENTS.md 第一段，不要埋到很后面。

3第三步：建记忆层

做什么

创建三个记忆文件：`USER.md`、`MEMORY.md`、`SOUL.md`。

为什么

聊天记录会断，模型会换，但文件能留下。记忆层就是把重要事实从对话里拿出来，放进长期可读的地方。

怎么做

mkdir D:\workspace\记忆层
notepad D:\workspace\记忆层\USER.md
notepad D:\workspace\记忆层\MEMORY.md
notepad D:\workspace\记忆层\SOUL.md
# USER.md

- 用户怎么称呼：
- 用户偏好：
- 常用路径：
- 不喜欢的做事方式：
# MEMORY.md

- 环境配置：
- 常用工具：
- 项目事实：
- 已验证经验：
# SOUL.md

- Agent 名字：
- 说话风格：
- 行为边界：
- 和用户的协作方式：
✅ 自检：三个文件都存在，打开后中文不乱码。
出错怎么办：Agent 记不住时，不要只说“你记住”。要把事实写进对应文件；如果乱码，用 VS Code 右下角编码改成 UTF-8 保存。

4第四步：双 Agent 协作

做什么

把 `当前状态.md` 当成人控总控台，而不是普通记事本。马哥负责总控和决策，小墨负责分析/方案/复盘，阿珂负责执行/修改/验证。

为什么

两个独立 Agent 不应该靠口头记忆传话。`当前状态.md` 是唯一任务交接面：谁负责、等谁决策、改了哪些文件、验证到哪一步，都写在这里。

怎么做

mkdir D:\workspace\执行流转层
notepad D:\workspace\执行流转层\当前状态.md

当前状态.md 总控台模板

# 当前状态

任务 ID：TASK-YYYYMMDD-001
任务名称：[任务名]
当前阶段：待分析 / 分析中 / 待决策 / 待执行 / 执行中 / 待验收 / 已完成
当前负责人：[小墨/阿珂/马哥]
验收人：马哥
依赖任务：无
最后更新：YYYY-MM-DD

## 用户原始目标
[马哥说了什么]

## 小墨分析结论
[方案摘要]

## 马哥决定
采用：[是/否]
不采用：[理由]
暂缓：[理由]

## 给阿珂的执行任务
[目标/修改位置/具体改动/不能改什么/验收标准]

## 阿珂执行结果
[改了什么/验证结果]

## 待确认问题
[阻塞/需要马哥看]

## 下一步
[具体动作]

## 涉及文件
- [路径1]
- [路径2]

状态流转

正常流转：待分析 → 分析中 → 待决策 → 待执行 → 执行中 → 待验收 → 已完成
异常流转：待决策 → 暂缓
异常流转：待验收 → 需要修改 → 待执行

小墨交付格式

第一层：给马哥看
- 判断：方向对/不对/砍掉什么
- 建议：怎么做
- 风险：哪里可能翻车
第二层：给阿珂的纯执行包
- 目标
- 修改位置
- 具体改动
- 不能改什么
- 验收标准
✅ 自检：新开一个对话，只给 Agent 读 `当前状态.md`，它能说出当前阶段、当前负责人、马哥决定、给阿珂的执行任务和下一步。
出错怎么办：如果交接混乱，先看当前阶段是不是停在“待决策”或“待验收”；如果没有马哥决定，不要让执行 Agent 直接改文件。

5第五步：加执行脑协议

做什么

创建一份专门约束执行风格的协议，让 Agent 不要闷头乱干。

为什么

`AGENTS.md` 管总规则，执行脑协议管具体做事风格：先审后干、敢砍简化、写完必验、成本意识。

怎么做

mkdir D:\workspace\规则层\模型规则
notepad D:\workspace\规则层\模型规则\执行脑协议.md

协议最小模板

# 执行脑协议

1. 先审后干：读完方案先判断是否可行。
2. 敢砍简化：过度设计直接压成 MVP。
3. 写完必验：代码、配置、文档都要检查。
4. 成本意识：能本地做不用 API。
5. 诚实原则：卡了就说卡在哪里。
6. 状态同步：项目结束更新 当前状态.md。

挂到 AGENTS.md

# 在 AGENTS.md 增加一行
- 执行任务前读取 D:\workspace\规则层\模型规则\执行脑协议.md。
✅ 自检：打开 `AGENTS.md`，能看到执行脑协议路径。
出错怎么办：Agent 没按协议执行时，把协议路径放到 AGENTS.md 前 10 行；太靠后容易被长上下文淹没。

6第六步：自动备份

做什么

把 workspace 定时备份到 GitHub 私有仓库，防止规则、记忆、状态文件丢失。

为什么

规则层和记忆层一旦丢了，Agent 就会“失忆”。备份是这套系统的保险。

怎么做

# 1. 在 GitHub 创建私有仓库，例如 workspace-backup
# 2. 本地初始化
cd D:\workspace
git init
git remote add origin https://github.com/你的用户名/workspace-backup.git

# 3. 第一次提交
git add .
git commit -m "Initial workspace backup"
git push -u origin main

任务计划程序

# 每天 00:00 运行备份脚本，示例：
schtasks /create /tn WorkspaceBackup /tr "D:\备份\backup_workspace.bat" /sc daily /st 00:00
注意：备份前必须排除 `.env`、Token、API Key、聊天记录、数据库和个人隐私文件。
✅ 自检：GitHub 私有仓库能看到提交；本地 `git status` 没有意外文件。
出错怎么办：中文路径乱码时，bat 只做启动器，逻辑放 ps1；push 失败时，检查 remote 地址和 GitHub 登录；误提交密钥时立刻撤销并换 key。

7第七步：配置自己的报告模板

做什么

先写自己的报告规则，再做一个单文件 HTML 骨架。不要照抄别人的模板，把输出标准固定下来。

为什么

报告模板的价值不是好看，而是固定输出标准：标题、摘要、步骤、验证、排错、结论。

怎么做

$Workspace = "D:\workspace"
$ReportRules = @(
  "# 报告规则",
  "",
  "- 单文件 HTML，CSS/JS 内嵌。",
  "- 主色固定一个，例如湖泊色 #2E8B8B。",
  "- 左侧导航，正文卡片。",
  "- 每份教程必须有：做什么 / 为什么 / 怎么做 / 自检 / 排错。",
  "- 不写入密钥、Token、隐私聊天记录。"
)
$ReportRules | Set-Content -Encoding UTF8 "$Workspace\规则层\报告规则.md"

HTML 骨架方向

文件放这里：你的盘:\workspace\规则层\报告模板.html

最小结构：
- nav：目录
- main：正文
- section：每一步
- style：内嵌 CSS
- script：只放必要交互
✅ 自检：复制模板后改一份新报告，本地浏览器能打开，打印预览能正常阅读。
出错怎么办：页面空白时看控制台；图表不显示时检查 CDN；移动端乱时先删复杂布局；不要把模板改成重型前端项目。

8第八步：验证整套系统

做什么

不要只看文件有没有创建，要验证 Agent 是否真的会按这套系统工作。

怎么做

# 让 Agent 做一个小任务，例如：
请按 D:\workspace\规则层\执行规则.md，生成一份今天的任务总结报告。

检查清单

启动能读规则

Agent 开始任务前会提到执行规则或当前状态。

过程能交接

`当前状态.md` 能看出做到哪一步。

错误能沉淀

踩坑会写进记忆或 BugTracker，而不是口头说说。

结果能发布

HTML 报告能本地打开，也能推到 GitHub Pages。

✅ 自检：关掉当前对话，新开一个，让 Agent 只读 `当前状态.md`，它仍然知道下一步。
出错怎么办：如果恢复失败，说明当前状态写得太虚；如果报告打不开，先本地验证再推送；如果 Agent 又开始乱猜，把失败写进记忆层或 BugTracker。
