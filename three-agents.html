教程 02 · Workflow

三 Agent 协作闭环

你会把“分析、执行、审查”拆成互斥角色，再用 Harness 文件把它们串起来。Agent 负责工作，人负责授权和最终判断。

理解模型下一篇：项目实战

核心模型

三种能力，不等于三个自动决策者

这套体系的关键不是 Agent 数量，而是权力分离。方案 Agent 负责想清楚，执行 Agent 负责做出来，审查 Agent 负责看成品；任何阶段都不能越过用户。

用户提出需求与约束
方案 Agent分析、方案、风险
执行 Agent实现、测试、部署
审查 Agent视觉只读审查
用户验收或决定返工
一句话原则审查 Agent 是眼睛，不是手；方案 Agent 负责判断，不负责执行；执行 Agent 负责落地，不负责最终裁决。

Step 1

写死角色边界

角色负责禁止写入位置
方案 Agent
方案脑		读上下文、澄清需求、出方案、拆风险、复盘修改代码、部署、删除文件、越过审批Harness/方案交换/
执行 Agent
执行脑		代码、文件、调试、构建、真实验证、部署擅自扩大范围、代替用户验收、直接根据审查报告返工项目目录 + 执行报告
审查 Agent
审查脑		看图、视频、UI、网页、PPT，输出问题与建议执行命令、改文件、部署、直接指挥执行 AgentHarness/审查记录/
用户
总控		授权、定方向、触发审查、最终验收无Harness/当前状态.md
边界不是礼貌建议如果审查 Agent 可以直接改文件，它既出题又答题；如果执行 Agent 可以自行验收，质量闸门就失效。

Step 2

按任务类型分派，不让所有任务都走同一条线

任务类型推荐流程是否需要审查 Agent
视觉产出方案 Agent → 执行 Agent → 审查 Agent → 用户验收用户手动触发
代码工程方案 Agent → 执行 Agent 实现与测试 → 用户验收有 UI 成品时可触发
纯知识方案 Agent 整理 → 用户验收不需要
独立视觉审查用户触发 → 审查 Agent 读材料 → 输出报告本身就是审查任务

分派结果写进任务卡的 `任务类型` 字段。不要因为有三个 Agent，就强迫每个小任务都经过三次转手。

Step 3

跑一遍完整闭环

1. 用户派活

说明目标、约束和成功标准，当前状态进入“待分析”。

2. 方案 Agent 出方案

方案写入 `Harness/方案交换/任务ID.md`，状态改为“待用户审批”。

3. 用户批准

明确说“采用/执行”，状态进入“待执行”，负责人切换为执行 Agent。

4. 执行 Agent 执行

按方案修改项目、跑测试、保存证据，执行报告写回方案交换。

5. 用户触发审查 Agent

只有视觉任务需要，并且必须由用户明确要求“审查”。审查 Agent 只读材料并写报告。

6. 用户验收

选择完成、返工、暂缓或驳回。完成后任务进入归档。

待分析 → 待决策 → 待执行 → 执行中 → 待审查 → 待验收 → 完成 → 归档
                         ↘ 阻塞 ↗

规则：没有用户明确批准，不得进入“待执行”。

Step 4

每次交接都带证据

“做完了”不是状态。执行报告至少包含产出路径、验证记录、已知问题和审查重点，让下一个角色不需要重新猜。

# WEB-DOCS-001 执行报告

## 产出
- index.html
- assets/site.css

## 验证
- HTML 解析：通过
- 390px 移动端：无横向溢出
- 键盘导航：通过

## 已知问题
- GitHub Pages 缓存可能延迟 1-2 分钟

## 待审查要点
- 首页层级是否清楚
- 三篇教程是否容易找到
方案证据目标、范围、假设、风险与成功标准。
执行证据修改文件、命令输出、截图和性能数据。
审查证据问题位置、原因、建议和结论等级。
验收证据用户明确决定完成、返工或暂缓。

Step 5

审查 Agent 只做视觉验收

审查 Agent 只在网页、视频、图片、PPT、海报等视觉产出中介入。纯代码、配置和文档逻辑默认不进入视觉审查流程。

审查维度检查内容
构图与层级主体、对齐、安全区、视觉重心
色彩与品牌设计系统一致性、强调色克制、禁用色
文字字号、截断、乱码、标题层级和可读性
动效与节奏曲线、快慢变化、同时运动数量、卡顿
信息表达三秒内能否理解、事实是否准确、有无误导
交付风险分辨率、编码、黑帧、音画同步、移动端
审查报告不是任务单执行 Agent 不能看到“需小修”就直接修改。必须先由用户确认，或由方案 Agent 整理成新的返工方案。

Step 6

返工也要重新过闸门

执行 Agent 产出初版
  → 用户触发审查 Agent
  → 审查 Agent 写审查报告
  → 用户决定返工
  → 方案 Agent 把问题整理成优化方案
  → 用户批准
  → 执行 Agent 二次修改
  → 审查 Agent 复审
  → 用户验收

这样做看起来多一步，却能避免“审查建议在执行时被随意解释”。每次返工都有明确范围，不会越改越大。

什么时候不要自动化只有当三 Agent 稳定跑过至少十个完整项目、手工转接成为明确瓶颈，再考虑 App 通知和半自动调度。先让流程从真实痛点里长出来。

验收

怎么证明协作闭环真的成立

角色没有越权方案 Agent 没改代码，审查 Agent 没执行修改。
状态没有跳步待决策必须经过用户批准才能进入执行。
切换不丢上下文新 Agent 只读任务卡与证据就能继续。
返工有新方案审查报告先被裁决，再进入二修。

完成标准 用户只需要做三件事：派活、审批、验收；其余信息通过 Harness 文件自然流转，不需要重复口头转述。
