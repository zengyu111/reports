Gate 0 · 需求确认

需求不清，不进入方案

在拆任务、选工具和写代码之前，先确认目标、现有输入、交付物和验收标准。方案越精确，执行 Agent 越不需要猜，成品质量越稳定。

直接反问缺少关键约束时，只问当前最影响方向的一个问题。
给出多方案用户还没定方向时，提供少量互斥方案并说明取舍。
反推参考有样片、截图或现成产品时，先提取结构、节奏和视觉规律。
冻结验收口径把“做到什么算完成”写进任务卡，再允许进入执行。
管道先确认什么交付前必须验证
视频受众、时长、画幅、素材、节奏、声音分镜连续、字幕安全区、音画同步、导出参数
图片用途、尺寸、主体、风格、参考、禁改区域主体一致、文字准确、边缘干净、目标尺寸
网页目标、信息架构、视觉方向、交互、终端真实浏览器、响应式、关键链路、控制台与构建
代码复现条件、输入输出、影响范围、成功标准最小修复、针对性测试、回归检查、代码审查
采集来源、权限、范围、频率、字段、去重规则覆盖率、增量边界、重复数据、失败记录

准备工作

先定义 Harness 不做什么

Harness 是调度协议壳，只回答五个问题：谁、何时、做什么、做到哪、谁审查。它不是新的代码仓库，也不是第二套知识库。

内容应该放哪里原因
项目源码代码存放处/<项目名>/项目可独立构建、测试和发布
长期知识workspace/知识库/与任务状态分离，便于复用
任务调度workspace/Harness/只保存活动状态和交接证据
运行台账workspace/执行流转层/保留历史记录，不再承担活动总控
关键边界不要把项目文件搬进 Harness，也不要让 Harness 自动绕过用户审批。

Step 1

创建目录结构

下面命令使用 PowerShell。把 `$Workspace` 改成你的实际路径。

$Workspace = "C:\workspace"
$Harness = Join-Path $Workspace "Harness"

@(
  $Harness,
  (Join-Path $Harness "任务队列"),
  (Join-Path $Harness "方案交换"),
  (Join-Path $Harness "审查记录"),
  (Join-Path $Harness "归档")
) | ForEach-Object {
  New-Item -ItemType Directory -Path $_ -Force | Out-Null
}
C:\workspace\Harness\
├── README.md          # 协议、字段、优先级
├── 当前状态.md         # 唯一活动事实源
├── 任务队列\           # 待决策、执行中、阻塞任务卡
├── 方案交换\           # 方案与执行报告
├── 审查记录\           # 只读审查证据
└── 归档\              # 完成、暂缓、驳回任务
自检五个子目录存在，并且 Harness 之外的项目目录没有被移动。

Step 2

写一份最小协议

`README.md` 的任务不是介绍品牌，而是定义所有 Agent 都必须理解的合同。至少写清任务 ID、核心字段、状态流转和冲突优先级。

# Harness 协议

Harness 只管理任务调度，不保存项目源码和长期知识。

## 核心字段
- 任务 ID
- 任务名称
- 任务类型
- 当前阶段
- 当前负责人
- 依赖任务
- 验收人

## 状态流转
待决策 → 待执行 → 执行中 → 待验收 → 完成 → 归档

没有用户明确批准，不得从“待决策”进入“待执行”。

## 冲突优先级
1. 用户当前明确指令
2. 安全边界与 AGENTS.md
3. Harness 协议
4. 单个任务卡
为什么先写协议目录只解决“放哪里”，协议才解决“谁能写、什么时候写、写了算不算”。

Step 3

建立当前状态与任务卡

任务 ID 在整个生命周期内不变。视觉任务额外记录审查状态，代码任务则记录测试证据。

# 当前状态

任务 ID：WEB-DOCS-001
任务名称：教程站升级
任务类型：代码工程
当前阶段：待决策
当前负责人：用户
上一个负责人：方案 Agent
依赖任务：Harness/方案交换/WEB-DOCS-001.md
验收人：用户

## 已完成
- 已输出站点方案

## 下一步
- 用户批准后交给执行 Agent

## 阻塞
- 无
阶段允许动作退出条件
待决策补充方案和风险用户明确批准、暂缓或驳回
执行中修改项目、测试、记录证据交付物和验证完成
待审查审查 Agent 只读检查报告落到审查记录
待验收用户看结果并裁决完成或返工

Step 4

把入口挂到每个 Agent

每个 Agent 的启动文件必须指向同一份 `Harness/README.md` 与 `Harness/当前状态.md`。角色规则可以不同，事实源不能不同。

用户派活、审批、验收
方案 Agent读状态，写方案
执行 Agent读批准方案，落地验证
审查 Agent只读产出，写报告
用户最终裁决或返工
# AGENTS.md 启动入口示例

- 开始任务先读 C:\workspace\Harness\README.md
- 再读 C:\workspace\Harness\当前状态.md
- 只执行已批准且负责人匹配自己的任务
- 完成后写验证证据并更新状态

Step 5

从旧状态迁移到 Harness

迁移不要直接移动或删除。先复制旧状态，切换所有读取入口，确认内容一致后，把旧文件降级为兼容快照。

  1. 复制旧 `当前状态.md` 到 `Harness/当前状态.md`。
  2. 更新 AGENTS、恢复脚本和状态校验脚本的读取路径。
  3. 对比新旧文件哈希，确保迁移过程没有改内容。
  4. 将旧文件标记为“兼容快照，不再是活动事实源”。
  5. 稳定运行后再由用户决定是否归档，Agent 不自行删除。
Get-FileHash "C:\workspace\执行流转层\当前状态.md" -Algorithm SHA256
Get-FileHash "C:\workspace\Harness\当前状态.md" -Algorithm SHA256
迁移失败的典型原因只复制了文件，却忘了修改恢复脚本；结果新会话仍读取旧路径，形成双事实源。

Step 6

用一次烟雾测试证明正式接管

不要只检查目录存在。创建一个极小任务，完整跑过方案、批准、执行、审查和归档。

入口统一所有 Agent 的 Harness 路径均可读，活规则不再引用旧状态。
脚本通过状态校验与会话恢复脚本读取新路径并正常退出。
哈希一致兼容期的新旧状态内容一致，避免迁移丢字段。
闭环成立执行报告和审查报告进入正确目录，任务最终归档。
python C:\workspace\scripts\validate_state.py
python C:\workspace\scripts\session_recovery.py
真实验证结果本体系用 `HARNESS-SMOKE-001` 验证：三 Agent 入口、两个 Python 脚本和新旧状态 SHA-256 全部通过。

排错

常见问题

现象根因处理
新会话仍读旧状态AGENTS 或恢复脚本路径未切换全局搜索旧路径，逐一确认活规则引用
两个 Agent 同时改当前状态没有负责人和阶段闸门任务卡写明当前负责人，只允许负责人推进
审查报告直接变成修改指令审查权与执行权混在一起报告先交用户裁决，再生成返工方案
Harness 越来越像杂物箱源码、知识、状态没有隔离只保留调度字段和证据链接，内容回到对应目录

完成标准 新开一次会话，只读 Harness 协议与当前状态，也能准确说出任务、负责人、已完成、阻塞和下一步。
